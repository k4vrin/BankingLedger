# ADR: Phase 5 Transaction Isolation And Account Locking

## Status

Accepted.

## Context

Transfer creation performs a read-check-write sequence on cached account balances. Concurrent transfers that debit the same account must not both validate against the same stale balance, and transfers touching the same account pair must not deadlock.

The application uses Oracle Database. Oracle provides statement-level read consistency, but that alone does not protect balance validation followed by balance mutation. The transfer path therefore needs explicit write serialization around the affected account rows.

## Decision

Transfer creation uses pessimistic account row locks through `AccountRepository.findByIdForUpdate`.

The locking rules are:

- Load both transfer accounts before balance validation.
- Acquire locks in deterministic account-id order.
- Preserve source and destination role mapping after ordered loading.
- Use the Oracle lock timeout generated by the JPA lock timeout hint.
- Re-read the idempotency record after locks are acquired and before writing transfer or ledger rows.

The project does not add optimistic retry behavior for transfer creation in Phase 5. Account entities still keep their `@Version` column as a secondary guard, but the selected transfer strategy is pessimistic serialization. Retrying optimistic conflicts would add a second concurrency model without improving the primary transfer path.

## Consequences

Concurrent transfers that touch the same account wait behind the transaction currently holding the row lock. After the first transaction commits, the next transfer validates against the newly committed balance.

Concurrent cross transfers such as `A -> B` and `B -> A` acquire locks in the same account-id order, so they do not form a circular wait.

If the row lock cannot be acquired before the configured timeout, Spring/JPA lock exceptions are mapped to a structured `409 Conflict` response with code `CONCURRENT_TRANSFER_CONFLICT`. The failed request is retryable and must not commit transfer, ledger, posting, idempotency, audit, or outbox rows.

If an optimistic version conflict still reaches the API boundary, it is also mapped to structured `409 Conflict` with code `CONCURRENT_TRANSFER_CONFLICT`.

Idempotency and locking solve different problems:

- Idempotency prevents duplicate money movement for client retries.
- Account locks protect balance validation and mutation.

The transfer use case checks idempotency before locking for normal replay, and checks it again after lock acquisition. The second check closes the race where a duplicate request starts before the original commits, waits on account locks, and should replay the completed response instead of failing on duplicate transfer data.

## Verification

Phase 5 tests cover:

- `findByIdForUpdate` loads accounts and enforces Oracle lock timeout behavior.
- Deterministic multi-account lock ordering.
- Sequential baseline transfer behavior.
- Independent concurrent transfers.
- Same-source concurrent transfers within available balance.
- Same-source concurrent transfers exceeding available balance.
- Same account pair concurrent transfers.
- Cross transfers between two accounts.
- Concurrent duplicate idempotency requests with the same payload.
- Concurrent duplicate idempotency requests with different payloads.
- Concurrent duplicate external references.
- Structured `409 Conflict` responses for pessimistic and optimistic lock failures.
